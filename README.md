# mini-dokploy

A self-hosted deployment platform. Provide a Git repo URL + Dockerfile path and it builds, pushes, and exposes your app on a generated subdomain — all from a browser UI.

---

## Setup

### Prerequisites

- **Docker Desktop** (or Docker Engine with Swarm support)
- **Node.js 20+**
- **`make`** (optional — `npm run up` / `npm run down` work without it)

### Start

```bash
make up
# or, if make is unavailable:
npm run up
```

Generates a `.env` with a random auth secret, initialises a single-node Swarm if needed, then builds the mini-dokploy image, pushes it to the local registry, and deploys everything as Docker Swarm services.

| URL | What |
|---|---|
| http://app.127.0.0.1.sslip.io | Mini-Dokploy UI |
| http://localhost:8080 | Traefik dashboard |
| http://dep-{id}.127.0.0.1.sslip.io | Your deployed apps |

### Stop

```bash
make down
# or:
npm run down
```

### Environment variables

| Variable | Default | Notes |
|---|---|---|
| `BETTER_AUTH_SECRET` | auto-generated by `make` | Must be set in production |
| `DATABASE_PATH` | `./data/mini-dokploy.db` | SQLite file path |
| `REGISTRY_HOST` | `127.0.0.1:5000` | Local registry address |
| `SWARM_NETWORK` | `mini-dokploy_traefik-net` | Overlay network name |
| `PORT` | `3000` | App listen port |

### Troubleshooting

**`docker push` fails with "server gave HTTP response to HTTPS client"**
Add `127.0.0.1:5000` to Docker's insecure registries. Docker Desktop: Settings → Docker Engine:
```json
{ "insecure-registries": ["127.0.0.1:5000"] }
```
Restart Docker after saving. (Newer Docker Desktop versions and Linux Engine don't need this.)

## Architecture

```
Browser
  │
  ├─ HTTPS/WS ──► Traefik (port 80)
  │                  │  routes by Host() header
  │                  ├─► mini-dokploy app (port 3000)
  │                  └─► dep-{id} services (any internal port)
  │
  └─ tRPC mutations ──► Next.js API routes
                           │
                           ├─ Drizzle ORM ──► SQLite (./data/*.db)
                           │
                           └─ setImmediate ──► Deploy Pipeline
                                                  │
                                                  ├─ simple-git: git clone --depth 1
                                                  ├─ Dockerode: docker build
                                                  ├─ Dockerode: docker push → local registry
                                                  └─ Dockerode: swarm service create/update
                                                       └─ Traefik picks up labels automatically

Live log streaming:
  Deploy pipeline ──► emitLog() ──► globalThis.__logBuses (EventEmitter)
                                         │
                                    WebSocket handler (ws/handler.ts)
                                         │
                                    Browser (WS connection)
```

### Key design choices

**Single custom HTTP server** (`src/server.ts`) — Next.js doesn't support WebSocket upgrades natively. The custom server handles the `/ws/logs` upgrade path and delegates everything else to Next.js.

**`globalThis.__logBuses`** — Next.js webpack bundles API routes and server code into separate module contexts. A plain `Map` would produce two isolated instances that can't see each other's events. Pinning to `globalThis` gives both the deploy pipeline and the WS handler a shared reference.

**WS subscribes before auth** — The handler registers its `onLog` listener *before* the async auth/DB checks, then buffers events received during that gap. Without this, events emitted between the DB fetch and subscription setup are silently dropped.

**Swarm VIP, no published ports** — Each deployed service gets a Virtual IP on the `traefik-net` overlay network. Traefik connects directly to `vip:port` — no host port is published. This means two apps can both listen on port 3000 internally with zero conflict.

**`PORT` env var injected automatically** — Every container receives `PORT=<exposedPort>` so apps that read `process.env.PORT` bind to the right port without configuration.

## Tradeoffs

### What's deliberately limited

| Decision | Why | Cost |
|---|---|---|
| SQLite over Postgres | Zero external dependencies, one-command start | Single writer; concurrent deploys can race on shared rows |
| In-process EventEmitter over Redis Pub/Sub | No broker to run or configure | Log streaming breaks if you run more than one app instance |
| `setImmediate` over a job queue | No queue infrastructure needed | Server restart mid-deploy leaves the deployment stuck in `building` forever |
| Docker Swarm over Kubernetes | Swarm is built into Docker, no cluster to provision | Less ecosystem, no pod autoscaling, no Helm charts |
| Local registry | No external registry credentials to manage | Images don't survive a `make down -v`; not suitable for multi-node |

### Scale ceiling

The current architecture is **single-node, single-process**. It handles a solo developer or small team comfortably. It starts breaking when:

- **Multiple simultaneous deploys** hit the same deployment ID — `seqRef` resets to 0 on each invocation, producing duplicate sequence numbers if two pipelines overlap.
- **Horizontal scaling** of the app container — the EventEmitter bus lives in process memory. WS clients on instance A miss events from a deploy running on instance B.
- **Long-running deploys + server restarts** — no job persistence means an interrupted build produces a phantom "building" status with no way to recover automatically.

### What I'd build next

- [ ] Encrypt env vars at rest — currently stored as plaintext JSON in SQLite
- [ ] Branch/tag selection for git clone — currently always clones the default branch
- [ ] Per-user deploy concurrency limit — prevent one user from saturating the build node
- [ ] Client-side WS reconnect — currently a dropped connection loses the live log tail
- [ ] Swap SQLite → Postgres + BullMQ for durable deploy jobs (survives restarts)
- [ ] Swap EventEmitter → Redis Pub/Sub for multi-instance log streaming
- [ ] HTTPS via Traefik Let's Encrypt — generated domains are HTTP-only today
- [ ] SSH key support for private repositories
- [ ] Rollback — redeploy from a previously pushed image tag without a rebuild
- [ ] Resource limits (CPU/memory) in the Swarm service spec — currently unconstrained
- [ ] Deployment versioning — keep a history of builds per project, not just the current state
- [ ] Custom domains — map a user-owned domain to a deployment

## How I used AI tools

AI was used as a force multiplier, not a replacement for judgment. The rule of thumb: delegate tasks where the search space is wide and the correctness signal is fast (the code runs or it doesn't); keep ownership over decisions where correctness is slow to verify and mistakes compound.

### What AI did well

**Navigating sparse APIs.** The Dockerode + Docker Swarm service spec surface is large and inconsistently documented. AI produced a working `CreateServiceOptions` shape, correct Traefik label structure, and restart policy values (in nanoseconds, as Swarm requires) without me having to cross-reference three different docs pages. Same for tRPC v11 + React Query v5 — the integration patterns are non-obvious and AI had them right.

**Subtle correctness under concurrency.** Two non-obvious patterns came out correct on the first pass:
- `ws/handler.ts` subscribes to live log events *before* the async auth check, buffering anything received during the gap — avoiding a race where events emitted between DB fetch and subscription setup are silently dropped.
- `globalThis.__logBuses` — Next.js webpack creates separate module contexts for API routes and server code, so a plain `Map` produces two isolated instances. AI correctly diagnosed this and suggested pinning to `globalThis`.

**Boilerplate that would have been slow to write, not hard to write.** Router patterns, Zod schemas, Drizzle queries, React form components — all idiomatic, nothing thrown away. This is where AI pays for itself most obviously.

### Where AI fell short

**Didn't reason about system-level conflicts.** The generated `EndpointSpec` published host ports (`PublishedPort: exposedPort`), which collapses at the OS level when two deployments use the same container port. AI produced something that works for one deployment and fails silently at two. Catching this required thinking about what Traefik actually needs (overlay VIP access) vs what the code was doing.

**Defaulted to `as any` instead of using real types.** `createService` was typed as accepting and returning `any` when `Dockerode.CreateServiceOptions` and `Dockerode.Service` exist and fit exactly. AI took the easy path. Fixed by reading `@types/dockerode` directly and letting the types flow through.

**Missed the edge case in the happy path.** When a WebSocket client connects to an already-finished deployment, the server replayed historical logs then closed — but never sent the `done` event the frontend uses to refresh the status badge. The main path (live deploy) was correct; this alternate path was silently broken.

**No production hygiene instinct.** `console.log` left in `logBus.emitLog` fires on every build event. Health check absent from the Swarm service spec — meaning Swarm can't distinguish a crashed container from a healthy one. Neither was flagged; both required manual review.

### What I kept for myself

**Schema and data model design.** How to cap log rows without blocking the pipeline, whether env vars and Docker labels should be separate columns, what statuses to track and what transitions to allow — these decisions cascade through the entire system and are slow to verify correct. They're also cheap to get wrong early and expensive to change later. Keeping them in one head rather than generating them prevents the subtle inconsistencies that emerge when structure is assembled piecemeal.

**The `seqRef: { n: number }` pattern.** A mutable reference shared across concurrent async callbacks to maintain a monotonic sequence counter. AI alternatives tended toward closures or classes — more syntax for no gain. Knowing when not to abstract is judgment, not generation.
